EXPERIMENT PROPOSAL: g188 Tokenizer-Flow Bridge

MOTIVATION:
g183 FAIL proved that corpus-derived PPMI SVD embeddings actively HURT training when used as anchor (-0.31 nats mean). Cross-tokenizer KD also HURTS (g180b: -0.37 to -0.54 nats). But within-family trained-anchor HELPS (+0.39 nats). The missing piece is a topology-aware bridge that maps trained embeddings from one tokenizer's codebook to another's.

CORE IDEA:
Instead of directly mapping tokens by string match (naive approach) or using raw co-occurrence statistics (PPMI SVD, which FAILED), use optimal transport on a bipartite token co-occurrence graph to create a soft mapping between tokenizer vocabularies. Then transfer trained embeddings through this mapping.

PROPOSED APPROACH:
1. Tokenize the same C4 corpus spans with both Qwen3 and GPT-2 tokenizers
2. Build bipartite co-occurrence graph: for each text span, which Qwen3 tokens co-occur with which GPT-2 tokens (positional alignment via character offsets)
3. Compute optimal transport plan between Qwen3 and GPT-2 token distributions
4. For each GPT-2 token, compute barycentric embedding from mapped Qwen3 trained embeddings
5. Use these "flow-bridged" embeddings as anchor target for a GPT-2-tokenizer Qwen3-arch model
6. Compare with: scratch, trained Qwen3 anchor (within-family, reference ceiling), direct string-match anchor, random-flow anchor (same OT structure but random source embeddings)

PASS CRITERIA (propose exact thresholds):
- Recover >= 0.10 nats vs scratch (20% of g181b +0.513)
- Beat direct string-match anchor by >= 0.05 nats (topology adds value)
- Beat random-flow anchor in 3/3 seeds (the OT mapping carries real information)
- Random-flow does not match flow-bridged (the source embedding quality matters)

COMPUTE ENVELOPE (from COMPUTE.md):

- Wall clock: <= 4h with checkpointing
- OT computation on 50K x 50K token matrix could be RAM-heavy — need sparse OT or subset
